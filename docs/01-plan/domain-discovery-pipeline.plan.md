# meta-pipe 기획 문서

> **요약**: 도메인 지식이 없어도, AI가 도메인을 조사하고 그 도메인에 최적화된 개발 파이프라인을 설계/실행하는 Claude Code 스킬. 사람과 자율 AI 에이전트 모두 사용 가능. \*\***프로젝트**: meta-pipe **버전**: 0.3 **날짜**: 2026-03-24 **상태**: Draft

---

## 1. 개요

### 1.1 목적

bkit 같은 기존 파이프라인은 "웹앱 개발"이라는 하나의 워크플로우에 고정된 9단계를 제공한다. 하지만 현실의 프로젝트는 도메인마다 필요한 단계가 다르다:

```
게임 개발     → 기획 → 레벨 디자인 → 프로토타입 → 게임플레이 루프 → 밸런싱 → 배포
핀테크 앱     → 규제 조사 → 보안 설계 → 스키마 → API → UI → 컴플라이언스 검증 → 배포
CLI 도구      → 기능 정의 → 커맨드 구조 → 구현 → 문서화 → 배포
Claude 스킬   → 목적 정의 → 트리거 설계 → SKILL.md 작성 → 테스트 → 배포
데이터 파이프라인 → 소스 분석 → ETL 설계 → 스키마 → 파이프라인 구현 → 모니터링 → 배포
```

meta-pipe는 이 문제를 해결한다:

1. 사용자가 "X를 만들고 싶다"고 말하면
2. AI가 X의 도메인을 조사하고
3. X에 최적화된 파이프라인 단계를 설계하고
4. 각 단계를 도메인 전문가 수준으로 진행한다

### 1.2 배경

**문제 1: 고정된 파이프라인은 범용적이지 않다**

```
bkit 9단계 = 웹앱에 최적화
  → "게임 만들고 싶어" → Phase 4(API 설계)가 맞지 않음
  → "CLI 도구" → Phase 3(목업), Phase 5(디자인 시스템) 불필요
  → "데이터 파이프라인" → Phase 6(UI 구현) 불필요
```

**문제 2: 도메인 지식이 없으면 파이프라인을 채울 수 없다**

```
일반 개발자 + "핀테크 앱"
  → 용어 정의 시도 → "KYC? AML? 원장?" → 모름
  → 전체 파이프라인 품질 저하
```

**해결: meta-pipe**

```
"핀테크 송금 앱 만들고 싶어" (도메인 지식 없음)
  ↓
[Step 1] 도메인 조사
  → 핀테크/송금 업계 용어, 규제, 표준 패턴 수집
  ↓
[Step 2] 파이프라인 설계
  → 이 도메인에 맞는 최적 단계 결정
  → 규제 조사 → 보안 설계 → 스키마 → ... → 컴플라이언스 검증 → 배포
  ↓
[Step 3] 단계별 실행
  → 각 단계를 핀테크 전문가처럼 진행
  → 사용자는 검토/의사결정만
```

### 1.3 리써치 결과 (2026-03-24)

| 조사 대상 | 결과 |
| --- | --- |
| 도메인 적응형 파이프라인 도구 | 존재하지 않음 |
| AI 요구사항 자동 수집 | IBM Watson 등 있으나, 인터뷰 기반이고 파이프라인을 설계하진 않음 |
| RAG 기반 도메인 학습 | 기존 문서가 있어야 함. 없는 지식을 조사하진 못함 |
| Claude Code 스킬 (192개+) | 도메인 조사 + 파이프라인 설계 기능을 가진 스킬 없음 |
| bkit 파이프라인 | 웹앱 고정 9단계. 도메인 조사 기능 없음. 단계 수 고정 |

→ **이 아이디어는 현재 시장에 없는 새로운 접근**

### 1.4 선행 사례 분석

| 프로젝트 | 핵심 개념 | meta-pipe와의 관계 |
| --- | --- | --- |
| [AWS AI-DLC](https://github.com/awslabs/aidlc-workflows) | 의도 분석 → 적응형 개발 워크플로우. greenfield/brownfield 감지, 복잡도에 따라 단계 깊이 조절. 마크다운 규칙 + 확장 시스템 | **가장 유사**. 적응형 워크플로우 패턴 참조. 단, 소프트웨어 개발 고정, 도메인 조사 없음 |
| [DataFlow Agent](https://github.com/OpenDCAI/DataFlow-Agent) | 의도 → 오퍼레이터 검색 → 동적 파이프라인 조립 (DAG) | 동적 파이프라인 생성 아키텍처 참조. 단, 데이터 처리 전용 |
| [Dify](https://github.com/langgenius/dify) (114K stars) | 비주얼 에이전트 워크플로우 빌더 | 범용 에이전트 플랫폼. 도메인 조사/파이프라인 설계는 안 함 |
| bkit 9단계 파이프라인 | 웹앱 고정 9단계 + PDCA | 출력 형식/PDCA 패턴 참조. 단, 고정 단계, 도메인 조사 없음 |

**차별점**: "도메인 조사 → 도메인 맞춤 파이프라인 동적 생성"을 하는 도구는 현재 없음.

## 2. 범위

### 2.1 포함

- [ ] **도메인 입력 수신**: 자유 텍스트로 프로젝트 아이디어 수용

- [ ] **도메인 조사**: WebSearch로 해당 도메인의 핵심 지식 수집

  - 업계 표준 용어, 규제/컴플라이언스, 표준 엔티티, 업계 패턴
  - 해당 도메인의 일반적인 개발 워크플로우

- [ ] **파이프라인 설계**: 도메인에 최적화된 단계를 동적으로 결정

  - 단계 수는 도메인에 따라 다름 (3\~12단계)
  - 각 단계의 목적, 산출물, 완료 기준 정의

- [ ] **단계별 실행**: 각 단계를 도메인 전문가 수준으로 진행

  - 각 단계에서 PDCA 사이클 적용
  - 산출물 문서 자동 생성

- [ ] **진행 상황 관리**: 현재 단계, 완료된 단계, 남은 단계 추적

- [ ] **자동 호출**: 사용자가 "X를 만들고 싶어" 등의 트리거 입력 시 meta-pipe가 자동 활성화

  - SKILL.md frontmatter `triggers` 필드로 구현
  - bkit 9단계로 커버 안 되는 도메인 감지 시 동적 파이프라인 생성

- [ ] **도메인 지식 캐시 (자동 개선 레벨 A)**: 실행 결과를 저장하여 동일 도메인 재요청 시 재사용/보완

  - Phase A 조사 결과를 `cache/{domain}/` 에 저장
  - 다음 실행 시 캐시 존재하면 WebSearch 스킵 가능 (보완만)
  - 실행 로그를 구조화하여 저장 (미래 메타-학습 데이터)

- [ ] **실행 로그 기록**: 모든 실행의 입력/출력/판단 근거를 로그로 남김

  - 미래 자율 AI 에이전트가 분석하여 파이프라인 로직 자체를 개선하는 데 사용
  - v1.0에서는 기록만, 분석/개선은 v2.0 (자율 에이전트) 범위

### 2.2 사용자 유형

이 파이프라인은 두 종류의 사용자가 사용한다:

|  | 사람 (Human) | AI 에이전트 (Autonomous Agent) |
| --- | --- | --- |
| 입력 | 자유 텍스트 대화 | 구조화된 JSON/마크다운 |
| 의사결정 | 직접 판단 | 규칙 기반 or 승인 요청 |
| 검토 | 매 단계 검토 가능 | approval_required 플래그에 따라 |
| 실행 | 대화형 (한 단계씩) | 자율 실행 (승인 불필요 단계는 연속) |

**설계 원칙:**

- 파이프라인 정의는 **이중 형식**: 사람이 읽는 마크다운 + AI가 파싱하는 구조화 메타데이터
- 각 단계에 `approval_required` 플래그: true면 사람 승인 후 진행, false면 자율 실행 가능
- AI-DLC 참고: 복잡도에 따라 단계 깊이 조절 (단순 작업 → 단계 축소)

### 2.3 제외

- 실제 코드 생성 (파이프라인과 문서만; 코드는 사용자/다른 스킬이 작성)
- 기존 코드베이스 리팩토링
- 인프라/배포 자동화 (문서 가이드까지만)

## 4. 핵심 설계 방향

### 4.1 "메타 파이프라인" 개념

```
일반 파이프라인 (bkit):
  고정 9단계 → 모든 프로젝트에 동일 적용

meta-pipe:
  입력 분석 → 도메인 조사 → 단계 설계 → 실행
  (파이프라인 자체가 동적으로 만들어짐)
```

### 4.2 3-Phase 핵심 구조

meta-pipe 자체의 워크플로우는 항상 3단계:

```
┌─────────────────────────────────────────────────────┐
│ Phase A: Discover (도메인 조사)                       │
│   입력: "X를 만들고 싶어"                              │
│   출력: 도메인 지식 보고서                              │
│   방법: WebSearch 병렬 조사                            │
│   ├─ "{domain} 핵심 용어 표준"                         │
│   ├─ "{domain} 규제 컴플라이언스"                      │
│   ├─ "{domain} 소프트웨어 개발 워크플로우 best practice"│
│   ├─ "{domain} 주요 엔티티 데이터 모델"                │
│   └─ "{domain} 일반적인 실패 사례 주의사항"            │
├─────────────────────────────────────────────────────┤
│ Phase B: Design Pipeline (파이프라인 설계)             │
│   입력: 도메인 지식 보고서                              │
│   출력: N단계 파이프라인 정의서                         │
│   각 단계 포함:                                        │
│   ├─ 단계 이름과 목적                                  │
│   ├─ 이 단계에서 할 일                                 │
│   ├─ 산출물 (파일명, 형식)                             │
│   ├─ 완료 기준                                         │
│   └─ 도메인 특화 주의사항                              │
├─────────────────────────────────────────────────────┤
│ Phase C: Execute (단계별 실행)                         │
│   입력: 파이프라인 정의서                               │
│   출력: 각 단계의 산출물 (문서들)                       │
│   방법: 단계 하나씩 순차 실행                           │
│   ├─ 현재 단계 안내                                    │
│   ├─ 도메인 전문가 수준 가이드                         │
│   ├─ 산출물 생성                                       │
│   ├─ 사용자 검토/승인                                  │
│   └─ 다음 단계로 이동                                  │
└─────────────────────────────────────────────────────┘
```

### 4.3 파이프라인 설계 예시

**입력: "핀테크 송금 앱"**

```
Phase A 조사 결과:
  - 핵심 규제: PCI-DSS, 전자금융거래법, AML/KYC
  - 핵심 용어: 원장, 정산, 충전, 이체, KYC, AML
  - 핵심 엔티티: 사용자, 계좌, 거래, 한도, 수수료

Phase B 설계 결과 (7단계):
  1. 규제/컴플라이언스 조사
  2. 용어 및 데이터 모델 정의
  3. 보안 아키텍처 설계
  4. 핵심 거래 플로우 설계
  5. API 설계 (결제 게이트웨이 연동 포함)
  6. UI/UX 설계 (KYC 인증 플로우 포함)
  7. 컴플라이언스 검증 체크리스트
```

**입력: "2D 인디 게임"**

```
Phase A 조사 결과:
  - 핵심 패턴: 게임 루프, ECS, 타일맵
  - 핵심 용어: 스프라이트, 콜리전, 렌더러, 씬
  - 핵심 엔티티: 플레이어, 적, 아이템, 맵, 레벨

Phase B 설계 결과 (6단계):
  1. 게임 컨셉/기획 (장르, 코어 루프)
  2. 기술 스택 결정 (엔진/프레임워크)
  3. 프로토타입 (코어 메카닉 검증)
  4. 레벨/콘텐츠 설계
  5. 밸런싱/플레이테스트
  6. 빌드/배포 (플랫폼별)
```

**입력: "CLI 도구"**

```
Phase B 설계 결과 (4단계):
  1. 커맨드 구조 설계 (서브커맨드, 플래그, 인자)
  2. 핵심 로직 구현
  3. 에러 처리/도움말/문서화
  4. 패키지 배포 (npm/brew/cargo)
```

### 4.4 bkit과의 관계

```
meta-pipe ≠ bkit 대체
meta-pipe = bkit의 상위 레이어 (또는 독립 도구)

시나리오 1: 독립 사용
  사용자 → meta-pipe → 도메인 맞춤 파이프라인 → 실행

시나리오 2: bkit과 연동
  사용자 → meta-pipe → 도메인 조사 완료
    → "이 프로젝트는 웹앱이네요" → bkit 9단계 파이프라인 추천
    → 도메인 지식을 bkit Phase 1에 주입
```

| 출력 언어 | 고정 / 자동 감지 | 입력 언어 따름 | 범용성 |

### 7.4 강제 메커니즘: 단계적 접근 (bkit 분석 기반)

bkit은 hooks(bash 스크립트) + context injection으로 Claude의 행동을 유도한다:

```
bkit 메커니즘:
  SessionStart hook → "MUST use AskUserQuestion" context 주입
  PreToolUse hook   → Write/Edit 전 task 분류 + PDCA 가이드 주입
  PostToolUse hook  → Write 후 gap analysis 제안 주입
  Stop hook         → 에이전트 종료 시 match rate 기반 다음 단계 제안
```

**meta-pipe 전략: 2단계 접근**

```
v1.0 (PoC/MVP): SKILL.md만
  - 프롬프트 기반 동작 (참조 문서 지연 로딩)
  - approval_required는 프롬프트 지시로 구현
  - 장점: 단순, 빠른 검증, 플러그인 패키징 불필요
  - 한계: Claude가 지시를 무시할 수 있음

v2.0 (프로덕션): SKILL.md + Hooks
  - PreToolUse: 파이프라인 단계 외 파일 쓰기 방지
  - PostToolUse: 단계 완료 검증, 다음 단계 자동 제안
  - Stop: approval_required 단계에서 사용자 승인 강제
  - SessionStart: 진행 상태 파일 로드, 중단점 복구
  - 장점: 에이전트 자율 실행 시 안전장치, 단계 전환 강제
  - 필요: 플러그인 패키징 (hooks/ + scripts/)
```

**v2.0 전환 시점 판단 기준:**

- AI 에이전트 자율 실행이 필요해질 때
- approval_required 무시 사례가 발생할 때
- 멀티 세션 연속성이 중요해질 때

## 8. AI-DLC 분석 결과 (2026-03-24)

### 8.1 AI-DLC 아키텍처 요약

AWS AI-DLC(`awslabs/aidlc-workflows`)는 소프트웨어 개발에 특화된 적응형 워크플로우 시스템.

**2-tier 규칙 구조:**

```
aws-aidlc-rules/core-workflow.md     ← 컨트롤러 (항상 로드)
aws-aidlc-rule-details/
  ├── common/          ← 공통 규칙 (항상 로드)
  ├── inception/       ← Inception 단계 (조건부 로드)
  ├── construction/    ← Construction 단계 (조건부 로드)
  ├── operations/      ← Operations (미래 확장)
  └── extensions/      ← opt-in 확장 (2-file: *.opt-in.md + *.md)
```

**핵심 메커니즘:**

- **지연 로딩**: core-workflow가 "Load from X"를 지시 → 그때 로드. Context window 절약
- **Execute IF / Skip IF**: 마크다운으로 단계 실행 조건 명시
- **Adaptive Depth**: Minimal / Standard / Comprehensive 3단계 상세도 조절 (6요소 기반)
- **Workspace Detection**: 파일 시스템 스캔으로 greenfield/brownfield 자동 감지
- **State + Audit**: `aidlc-state.md`(체크박스) + `audit.md`(타임스탬프 로그)
- **Extensions opt-in**: 질문 파일 + 규칙 파일 쌍. 수락 시 매 단계 blocking constraint

### 8.2 meta-pipe 적용 패턴

| AI-DLC 패턴 | meta-pipe 적용 | 핵심 차이 |
| --- | --- | --- |
| 2-tier 규칙 (core + details) | SKILL.md(컨트롤러) + references/(지연로딩) | rule-details가 **런타임에 동적 생성** |
| Workspace Detection | → Domain Discovery. 기존 도메인 아티팩트 유무 감지 | 파일 스캔이 아닌 **WebSearch** |
| Adaptive Depth (3단계 상세도) | 도메인 복잡도에 따라 단계 수 + 깊이 조절 | 단계 **수 자체**가 달라짐 |
| Execute IF / Skip IF | 각 생성 단계에 실행 조건 포함 | 조건이 도메인 조사 결과에서 도출 |
| Extensions opt-in | 도메인별 규제/컴플라이언스 확장 | 확장이 **동적으로 생성/제안** |
| State + Audit | .meta-pipe-status.json + 단계별 로그 | 상태 스키마가 **유연** (동적 단계) |
| Question-in-file | approval_required 단계에서 파일 기반 Q&A | 에이전트 자율 응답도 처리 |

**핵심 차이**: AI-DLC는 "소프트웨어 개발" 도메인 지식을 하드코딩. meta-pipe는 도메인 지식 자체를 Phase A에서 동적 획득, Phase B에서 규칙 자동 생성.
