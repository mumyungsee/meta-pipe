# Phase B: Design Pipeline - 카드뉴스 자동화 도구 파이프라인

> **도메인**: 카드뉴스 자동화 도구 개발
> **설계 일자**: 2026-03-24
> **기반**: Phase A 도메인 조사 결과

---

## 파이프라인 개요

```
입력: "카드뉴스 자동화 도구를 만들고 싶어"
  ↓ Step 1: 도메인 용어 & 요구사항 정의
  ↓ Step 2: 데이터 모델 설계
  ↓ Step 3: 템플릿 엔진 설계
  ↓ Step 4: 콘텐츠 파이프라인 설계
  ↓ Step 5: 렌더링 엔진 구현
  ↓ Step 6: 플랫폼 어댑터 & 내보내기
  ↓ Step 7: 접근성 & 법적 준수 레이어
  ↓ Step 8: 통합 테스트 & 품질 검증
출력: 카드뉴스 자동 생성 도구 (텍스트 입력 -> 멀티 플랫폼 카드뉴스 이미지)
```

## Step 1: 도메인 용어 & 요구사항 정의

**목적**: 카드뉴스 도메인의 용어를 코드베이스 전체에서 일관되게 사용하고, 핵심 기능 범위를 확정한다.

**할 일**:
- 도메인 용어 사전 작성 (한/영 매핑)
- 지원 플랫폼 목록 확정 (인스타그램, 페이스북, 블로그, 카카오톡 등)
- 판형/해상도 매트릭스 정의
- 사용자 페르소나 정의 (마케터, 콘텐츠 크리에이터, 기업 홍보팀)
- MVP 기능 범위 확정

**산출물**:
- `docs/02-design/glossary.md` - 도메인 용어 사전
- `docs/02-design/requirements.md` - 기능 요구사항 명세

**완료 기준**:
- 모든 도메인 용어에 한/영 매핑 존재
- 지원 플랫폼별 사이즈 매트릭스 완성
- MVP vs Future 기능 구분 완료

**도메인 특화 주의사항**:
- "카드"와 "슬라이드"를 혼용하지 않도록 용어 통일
- 플랫폼별 사이즈 제약이 자주 변경되므로, 하드코딩 금지

**approval_required**: true
> 이유: 용어 정의와 기능 범위는 이후 모든 단계의 기초. 잘못되면 전체 재작업.

---

## Step 2: 데이터 모델 설계

**목적**: 카드뉴스의 모든 구성요소를 표현할 수 있는 데이터 구조를 설계한다.

**할 일**:
- 핵심 엔티티 정의 (Project, Slide, Element, Template, StyleGuide, ExportConfig)
- 엔티티 간 관계 정의
- JSON Schema 작성 (에이전트용 구조화 형식)
- 타입 시스템 설계 (TypeScript 타입 또는 JSON Schema)
- 마이그레이션 전략 (버전 관리)

**산출물**:
- `docs/02-design/data-model.md` - 데이터 모델 설계서
- `schemas/card-news.schema.json` - JSON Schema 정의
- `schemas/examples/` - 예시 데이터 파일

**완료 기준**:
- 모든 카드뉴스 구성요소가 모델로 표현 가능
- JSON Schema 유효성 검증 통과
- 최소 3개 도메인 예시(마케팅, 교육, 뉴스)로 모델 검증

**도메인 특화 주의사항**:
- Element의 다형성 처리: TextElement, ImageElement, ShapeElement 등
- 슬라이드 타입(cover/body/closing)별 필수 요소 차이 반영
- 접근성 필드(alt_text) 필수화
- 라이선스 정보(이미지, 폰트) 추적 필드 포함

**approval_required**: true
> 이유: 데이터 모델은 전체 아키텍처의 뼈대. 변경 비용이 매우 높음.

## Step 3: 템플릿 엔진 설계

**목적**: 재사용 가능한 카드뉴스 템플릿을 정의하고 관리하는 시스템을 설계한다.

**할 일**:
- 템플릿 구조 설계 (레이아웃 슬롯 시스템)
- 기본 레이아웃 유형 정의 (텍스트 중심, 이미지 중심, 하이브리드 등)
- 스타일 변수 시스템 (컬러 팔레트, 폰트셋을 변수로)
- 템플릿 상속/조합 메커니즘
- 슬라이드 타입별 기본 템플릿 (cover, body, closing)

**산출물**:
- `docs/02-design/template-engine.md` - 템플릿 엔진 설계서
- `templates/` - 기본 템플릿 정의 파일들

**완료 기준**:
- 최소 5개 기본 템플릿 정의 완료
- 스타일 변수 변경만으로 다른 분위기 생성 가능
- cover/body/closing 각각에 최소 2개 레이아웃 변형

**도메인 특화 주의사항**:
- Tyle.io의 29개 레이아웃 패턴 참고하되, 프로그래밍 방식에 맞게 추상화
- 텍스트 길이에 따른 자동 폰트 크기 조정 로직 필요
- 이미지 비율에 따른 자동 crop/fit 전략
- 플랫폼별 판형 차이를 템플릿 레벨에서 흡수

**approval_required**: false
> 이유: Step 2의 데이터 모델이 확정되면 템플릿 엔진은 구현 세부사항.

## Step 4: 콘텐츠 파이프라인 설계

**목적**: 원시 텍스트/주제에서 카드뉴스 콘텐츠(카피, 이미지 키워드, 슬라이드 구성)를 자동 생성하는 파이프라인을 설계한다.

**할 일**:
- 입력 형식 정의 (자유 텍스트, 구조화 데이터, URL 등)
- 텍스트 -> 슬라이드 분할 알고리즘 설계
- 카피라이팅 프롬프트 설계 (제목, 본문, CTA)
- 이미지 선택/생성 전략 (키워드 추출 -> 스톡 이미지 검색 or AI 생성)
- 스토리라인 자동 구성 로직

**산출물**:
- `docs/02-design/content-pipeline.md` - 콘텐츠 파이프라인 설계서
- `prompts/` - LLM 프롬프트 템플릿

**완료 기준**:
- 자유 텍스트 입력으로 슬라이드 구성 자동 생성 가능
- 생성된 콘텐츠가 카드뉴스 best practice 준수 (카드당 1메시지)
- 3가지 입력 유형(주제어, 블로그 글, 뉴스 URL)에서 작동

**도메인 특화 주의사항**:
- 카드당 텍스트량 제한 (제목 15자 이내, 본문 50자 이내 권장)
- 스토리 흐름의 자연스러움 검증 로직
- 표지는 호기심 유발, 마지막 장은 CTA 패턴 적용
- 이미지 키워드는 텍스트 내용과 의미적 일치 필요

**approval_required**: true
> 이유: 콘텐츠 품질이 최종 결과물의 핵심. LLM 프롬프트 설계는 반복 검증 필요.

## Step 5: 렌더링 엔진 구현

**목적**: 데이터 모델 + 템플릿 + 콘텐츠를 결합하여 실제 이미지를 생성하는 엔진을 구현한다.

**할 일**:
- 렌더링 방식 확정 (HTML/CSS -> Puppeteer 스크린샷 권장)
- HTML 템플릿 렌더러 구현
- 폰트 임베딩/로딩 시스템
- 이미지 합성 및 필터 처리
- 렌더링 품질 최적화 (DPI, 안티앨리어싱)

**산출물**:
- `src/renderer/` - 렌더링 엔진 소스 코드
- `docs/02-design/rendering-engine.md` - 렌더링 엔진 설계서

**완료 기준**:
- 데이터 모델 입력으로 이미지 출력 가능
- 출력 이미지가 1080px 기준 선명 (2x DPI 지원)
- 한글 폰트 렌더링 정상 (나눔스퀘어, 프리텐다드 등)
- 렌더링 시간: 슬라이드 1장당 3초 이내

**도메인 특화 주의사항**:
- **한글 폰트 렌더링**이 가장 큰 기술적 과제. Puppeteer에서 한글 폰트 경로 설정 필수
- 행간(line-height) 계산이 CSS와 디자인 도구에서 다를 수 있음
- 이미지 위 텍스트의 가독성 보장 (반투명 오버레이 등)
- Retina 디스플레이 대응 (2x 렌더링)

**approval_required**: false
> 이유: 기술 구현 단계. Step 2~4가 확정되면 구현 범위가 명확.

---

## Step 6: 플랫폼 어댑터 & 내보내기

**목적**: 생성된 카드뉴스를 각 SNS 플랫폼에 맞는 형식으로 변환하고 내보내는 시스템.

**할 일**:
- 플랫폼별 어댑터 구현 (사이즈, 포맷, 메타데이터)
- 배치 내보내기 (전체 슬라이드 한번에)
- 파일 네이밍 컨벤션
- 이미지 포맷 최적화 (PNG vs JPG vs WebP)
- [검토 필요] 플랫폼 API 직접 게시 기능 (인스타그램 Graph API 등)

**산출물**:
- `src/exporters/` - 플랫폼별 내보내기 모듈
- `docs/02-design/export-spec.md` - 내보내기 사양

**완료 기준**:
- 최소 3개 플랫폼 지원 (인스타그램 1:1, 페이스북 16:9, 스토리 9:16)
- 한 프로젝트에서 멀티 플랫폼 동시 내보내기 가능
- 출력 파일 용량 최적화 (인스타그램 제한: 30MB 이내)

**도메인 특화 주의사항**:
- 플랫폼별 이미지 사이즈 제약이 수시로 변경됨 -> 설정 파일로 관리
- 텍스트가 많은 카드는 PNG, 사진 중심 카드는 JPG가 유리
- 파일명에 슬라이드 순서 포함 (01_cover.png, 02_body.png ...)
- ZIP 묶음 내보내기 지원

**approval_required**: false
> 이유: 기술 구현 세부사항. 플랫폼 사양은 외부 문서 참조.

## Step 7: 접근성 & 법적 준수 레이어

**목적**: 카드뉴스의 접근성 문제와 저작권/라이선스 리스크를 시스템 레벨에서 방지한다.

**할 일**:
- 접근성 텍스트 자동 생성 (각 슬라이드의 alt text)
- 이미지 라이선스 추적 시스템
- 폰트 라이선스 검증 로직
- 접근성 보고서 생성 (텍스트 버전의 카드뉴스)
- 저작권 워터마크/출처 자동 삽입

**산출물**:
- `src/compliance/` - 접근성/법적 준수 모듈
- `docs/02-design/accessibility-spec.md` - 접근성 사양

**완료 기준**:
- 모든 슬라이드에 alt text 존재
- 사용된 이미지/폰트의 라이선스 정보 추적 가능
- 텍스트 전용 버전 자동 생성 (시각장애인 접근성)
- 라이선스 위반 이미지/폰트 사용 시 경고

**도메인 특화 주의사항**:
- 이미지 저작권 위반 시 3년 이하 징역/3천만원 이하 벌금 -> 반드시 시스템에서 방지
- 나눔스퀘어, 프리텐다드 등 한국 오픈소스 폰트 기본 제공
- Pexels, Unsplash 등 CC0 이미지 소스 연동
- WCAG 2.1 가이드라인 참조

**approval_required**: true
> 이유: 법적 리스크가 높은 영역. 라이선스 정책은 사람이 검증해야 함.

---

## Step 8: 통합 테스트 & 품질 검증

**목적**: 전체 파이프라인(텍스트 입력 -> 이미지 출력)을 end-to-end로 검증한다.

**할 일**:
- E2E 테스트 시나리오 작성 (3개 도메인: 마케팅, 교육, 뉴스)
- 시각적 회귀 테스트 (스냅샷 비교)
- 성능 벤치마크 (10장 카드뉴스 생성 시간)
- 카드뉴스 품질 체크리스트 자동 검증
- 사용자 피드백 수집 & 반영

**산출물**:
- `tests/e2e/` - E2E 테스트 코드
- `tests/snapshots/` - 시각적 스냅샷
- `docs/02-design/quality-checklist.md` - 품질 체크리스트

**완료 기준**:
- 3개 도메인 E2E 테스트 통과
- 렌더링 결과 시각적 일관성 확인
- 10장 카드뉴스 전체 생성 시간 30초 이내
- 품질 체크리스트 80% 이상 자동 검증

**도메인 특화 주의사항**:
- 카드뉴스 품질 체크: 가독성, 일관성, 여백, 폰트 크기, 이미지 품질
- 모바일 화면 시뮬레이션 필수 (375px 기준)
- 한글 특화 테스트: 줄바꿈, 자간, 행간, 장평
- 실제 SNS 업로드 후 렌더링 확인

**approval_required**: false
> 이유: 테스트/검증은 자율 실행 가능. 결과만 리포트.
