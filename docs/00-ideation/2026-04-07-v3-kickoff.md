## 1. v2 테스트에서 발견한 문제

### 1.1 Phase A (Consult) 결함

**문제 1 — 결과물 형태/스타일 수집 누락**

- v2 A1이 수집하는 것: 목표(`goal`), 자동화 수준, 사용 주체, 품질 기대치
- 수집하지 않는 것: "어떤 모습의 결과물을 원하는지" (스타일, 형식, 예시)
- 영향: 자동화 방식에 따라 결과물 스타일이 달라지는데, 사용자 선호를 모르니 경로 제안이 빗나감

**문제 2 — 경로 제안 테이블이 추상적**

- v2 A4 테이블: 결과물 수준 = `높/중/낮`
- 도메인 지식 없는 사용자에게 "높음"은 아무 의미 없음
- 각 경로에서 실제로 어떤 결과물이 나오는지 구체적 묘사가 없어 선택 불가

**문제 3 — 단계별 검수 불가**

- 도메인 모르는 사용자는 Phase B(도메인 조사), C(파이프라인 설계) 결과물도 검수 불가
- 설계가 사용자가 각 단계 산출물을 판단할 수 있다고 암묵적으로 전제함

### 1.2 설계 복잡도 문제

v2 설계에 포함된 기능 목록:

| 기능 | 비고 |
|------|------|
| Phase A~F 전체 흐름 | 기본 |
| 실행 모드 태깅 (auto/assist/manual) | 기본 |
| 강등 체인 (auto→manual) | 추가 |
| 업그레이드 메커니즘 (manual→auto) | 추가 |
| 사용자 프로필 시스템 | 추가 |
| 세션 재개 | 추가 |
| 캐시 시스템 | 추가 |
| Adaptive Depth (3~12단계) | 추가 |
| 복잡도 레벨 (Light/Standard/Full) | 추가 |
| Phase 간 인터페이스 JSON | 추가 |
| 컨텍스트 예산 관리 | 추가 |

Phase A에서 이미 문제가 나왔는데, 이 위에 전부 얹으면 어디서 문제가 생긴 건지조차 파악 불가.

## 8. v2에서 배운 개발 방법론 교훈

### 문제

v2에서 Phase A → B → C를 순서대로 구현했지만 Phase 사이에 테스트 없이 바로 다음 Phase로 넘어감.
다 만들고 테스트하니 Phase A에서 문제 발견 → B, C도 A에 의존하므로 전부 흔들림.

### v3 방향

- **각 Phase마다 구현 후 테스트** — Phase A 테스트 통과해야 Phase B 구현 시작
- **작은 단위로 잘라서 구현 + 테스트** — 바이브코딩은 생각보다 단위를 작게 가져가야 함
- **Plan 단계에서 테스트 방법까지 정의** — "완료 기준 + 테스트 방법"을 Phase마다 명시

### Phase A 테스트 기준 (예시)

| 테스트 항목 | 합격 기준 |
|------------|----------|
| 인터뷰 완성도 | 결과물 형태/스타일이 수집됐는가 |
| 경로 제안 | 도메인 모르는 사람이 읽고 차이를 구분할 수 있는가 |
| 선택 납득도 | 선택 후 "내가 뭘 선택했는지" 알 수 있는가 |
| consult-result.json | 필수 필드 다 있는가 |

테스트 방법: 실제 테스트 도메인으로 `/meta-pipe` 실행 → 사람이 직접 인터뷰 받아보기

### Plan 문서에 포함해야 할 것

각 Phase마다:
```
완료 조건:
  - 테스트 도메인 N개로 실제 실행
  - 산출물(JSON) 생성 확인
  - 사람이 읽고 판단 가능한지 확인
  → 통과해야 다음 Phase 구현 시작
```

테스트 도메인도 Plan 단계에서 미리 확정 필요.
