## 1. 회의 배경

v3 킥오프(2026-04-07) 후, Plan 문서 작성에 들어가려 했으나 "지금 이대로 계획을 세워도 되는가?"라는 근본적 의문이 제기됨. v1→v2→v3로 솔루션은 계속 다듬었지만, **"왜 이걸 만드는가"가 점점 흐려졌다**는 인식에서 출발.

## 2. 문제 정의 — 왜 v3 진입이 막혔나

### 사용자가 느끼는 혼란

1. **오픈소스 폭발**: Open Claw 등으로 바로 쓸 수 있는 파이프라인이 대량 공개 중. 검증된 파이프라인을 가져다 AI에게 커스터마이징 시키면 되는 것 아닌가?
2. **지피터스 사례**: 사례 게시글 기반으로 자동화 수업 도구를 이미 만듦. 사례 기반이면 "결과를 가늠할 수 없어서 선택이 어렵다"는 문제가 해결됨.
3. **반복되는 패턴**: 직접 만들면 완성도가 충분하지 않아 버려두고, 결국 남이 잘 만든 오픈소스를 쓰게 됨.

### 근본 문제

meta-pipe v1~v2의 접근은 **"처음부터 파이프라인을 생성"**하는 것. 하지만 사용자 본인의 실제 행동 패턴은 **"검증된 것을 찾아서 커스터마이징"**하는 것. 설계와 실제 행동이 불일치.

---

## 3. 핵심 질문 3가지와 답변

### Q1. 이걸 왜 만들고 싶어? 본인 문제야, 남을 위한 거야?

> **본인 문제**. 하지만 오픈소스가 쏟아지는 상황에서 직접 만들 필요가 있나 회의감이 있음. 그럼에도 만들어야 하는 이유는 **bkit 회사와 콜라보** 목표가 있기 때문.

### Q2. "이게 있으면 좋겠다"는 순간이 있어?

> 직접 답변 대신 행동 패턴으로 확인됨:
> - 파이프라인을 찾으면 AI로 커스터마이징은 쉬움
> - 문제는 **뭘 골라야 할지 선택 자체가 어려움** (도메인 없으니까)
> - 실행 결과를 **사전에 예측할 수 없음** (같은 파이프라인 사례를 보지 않는 한)

### Q3. v2까지 만들고 멈춘 이유?

> 직접 만들면 완성도가 기대에 못 미침 → 버려둠 → 나중에 남이 잘 만든 것을 쓰게 됨. 이 패턴의 반복.

## 4. A안 vs B안 — 방향 결정

| 구분 | A안 (기존) | B안 (피봇) |
|------|-----------|-----------|
| 접근 | AI가 파이프라인을 처음부터 생성 | 검증된 사례를 찾아서 적용 |
| 결과 예측 | 어려움 | 사례 결과로 예측 가능 |
| 사용자 행동과 일치 | ❌ | ✅ |
| 완성도 리스크 | 높음 (처음부터 만드니까) | 낮음 (검증된 것 기반) |

**결정: B안 채택.**

단, 인터뷰/컨설팅이 불필요해지는 건 아님. 오히려 **내게 맞는 사례를 찾기 위해** 더 정확한 인터뷰가 필요.

## 5. 사례 라이브러리 충돌 문제와 해결

### 문제

여러 소스(GitHub, Threads, 지피터스)에서 사례를 가져오면 방법론/스택 충돌 가능. 예: n8n 기반 사례와 Python 스크립트 기반 사례를 섞으면 작동 안 함.

### 해결 방향

코딩의 라이브러리 충돌 해결과 동일한 접근:
1. **Phase A에서 스택 제약 조건을 먼저 수집** — "어떤 도구 쓸 수 있어?", "유지보수 직접 할 수 있어?" 등
2. 제약 조건이 확정되면 **검색 범위가 자동 필터링**됨
3. 충돌 가능한 사례는 처음부터 제외

→ Phase A의 핵심이 변경됨: ~~"뭘 만들고 싶어?"~~ → **"어떤 제약 안에서 만들거야?" + "뭘 만들고 싶어?"**

## 6. meta-pipe의 존재 이유 (bkit 콜라보 맥락)

### 큰 그림

1. **meta-pipe를 bkit 플러그인으로 통합** → bkit을 도메인에 매이지 않고 폭넓게 사용 가능하게
2. **bkit + meta-pipe를 오픈클로 같은 자율형 AI 에이전트에 제공** → AI가 스스로 PDCA 사이클을 돌리며 자동화 도구를 만들고 사용
3. **다음 기수 스터디를 bkit 회사에 제안** → 수강생이 meta-pipe + bkit으로 나만의 자동화를 만듦
4. **스터디 산출물을 bkit 플러그인 스토어에 올려 생태계 확장**

### 부가 목표

- bkit 앰버서더 경로 — 앰버서더처럼 활동하다 보면 자연스럽게 연결
- 연간 목표(강의/유튜브)의 한 축으로 활용
- 안 되더라도 연간 목표를 계속 진행하는 것일 뿐

### 모든 과정이 문서로 남는 것의 가치

- bkit PDCA로 작업하면 과정 문서가 자동으로 쌓임
- 이 문서들이 **개인 지식베이스**가 됨 → 개인 온톨로지, 시스템 개선, 지식경영
- **모든 문서가 로컬 PC에 남아 온전한 내 소유** → RAG, LLM 학습 등 활용 무궁무진
- 맥락이 계속 보존되므로 결과가 쌓이며 발전

## 7. 지피터스 대비 차별점

| 항목 | 지피터스 | meta-pipe + bkit |
|------|---------|-----------------|
| 인터페이스 | GUI (사이트 접속) | CLI (bkit 플러그인, GUI에 갇히지 않음) |
| 확장성 | 서비스 내 제한 | 오픈클로 등 자율형 AI에 제공 가능 |
| 자가 개선 | ❌ | PDCA 사이클로 검증/개선 루프 |
| 문서화 | 사례 게시글 | 전 과정 자동 문서화 → 개인 지식베이스 |
| 소유권 | 플랫폼 종속 | 로컬 PC, 온전한 내 소유 |

**가장 강력한 차별점**: "만들고 끝"이 아니라 "만들고, 검증하고, 개선하는 루프" + 모든 과정이 문서로 남음

## 8. 검토 결과 — 보완/주의 사항

### 8.1 스코프 주의

v3의 유일한 사용자는 **"나"** 한 명. 스터디/앰버서더/커뮤니티 생태계는 v3 이후에 고려. 스코프가 커지면 v1, v2와 같은 패턴 반복 위험.

### 8.2 반복 패턴 경고

> "만들었다 해도 충분히 사용할 만큼 완성도가 있지 않아서 버려두고 있다가 다른 사람이 잘 만든 걸 쓴다"

이 패턴을 막으려면:
- **"사용 가능한 최소 버전"을 먼저 정의**
- **첫 번째 테스트를 빨리 실행** — 설계 2주보다 "대충 만들고 써보기 3일"이 나음

### 8.3 Phase B→C 사이 갭

사례를 찾았는데 내 환경에서 안 돌아가는 경우에 대한 판단 단계가 필요. "이미 실행된 사례"라도 그 사람 환경과 내 환경이 다를 수 있음.

### 8.4 검색 방식 결정

사용자가 직접 검색하는 건 비효율적. **사이트 지정 검색(site: 연산자)**으로 AI가 검색~요약까지 수행, 사용자는 선택만:
- `site:github.com` → 실행 가능한 코드/오픈소스 레포
- `site:threads.net` → 실제 사용 후기/팁
- `site:gpters.org` → 검증된 사례 + 실행 결과

## 9. v3 Phase 구조 (B안 기준)

| Phase | 이름 | v2 대비 변경 | 핵심 |
|-------|------|------------|------|
| A | Consult | **제약 조건 수집 추가** | 목표 + 제약(스택/도구/역량) + 결과물 형태 수집 |
| B | Search | **Discover에서 변경** | 지정 사이트에서 사례 검색 + 결과 예시 제시 |
| C | Adapt | **Design에서 변경** | 선택한 사례를 내 상황에 맞게 수정 |
| E | Execute | 동일 | 실행 |

### Phase 간 흐름

```
Phase A (Consult)
  → 목표 + 제약 조건 + 결과물 형태 수집
  → 검색 키워드 생성

Phase B (Search)
  → 지정 사이트별 검색 (WebSearch + site:)
  → 유망한 결과 상세 수집 (WebFetch)
  → 사례별 요약 — "이건 이런 결과가 나온다" 제시
  → 사용자 선택

[판단 단계: 선택한 사례가 내 환경에서 실행 가능한가?]

Phase C (Adapt)
  → 선택한 사례를 내 요구사항에 맞게 수정
  → 파이프라인화

Phase E (Execute)
  → 실행
```
