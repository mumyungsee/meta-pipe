## 1. 논의 배경

### v1의 한계

meta-pipe v1은 "도메인 조사 -&gt; 파이프라인 설계 -&gt; 실행"의 3단계(Phase A/B/C)로 구성되어 있었다. 실제로 youtube-thumbnail, card-news 두 도메인에 테스트 적용했으나 다음과 같은 문제를 발견:

- **도구 세팅 문제**: 프로그램 개발은 이미 라이브러리/프레임워크가 갖춰져 있어 코드만 짜면 되지만, 이미지/영상 등 비개발 도메인은 이미지 AI 선택(MCP? API? 로컬 LLM?), 세팅, 비용 결정 등 사전 작업이 많다.
- **"돌리면 뚝딱" 환상**: 파이프라인을 만들어도 도구 세팅 없이는 실행 불가. Make/n8n처럼 API 세팅이 필요한데, 그 과정이 생략되어 있었다.
- **사용자 맥락 부재**: 사용자가 뭘 원하는지, 어떤 도구를 쓸 수 있는지, 역량이 어느 정도인지 파악하는 과정이 없었다.

### bkit과의 근본적 차이

- **bkit (개발 파이프라인)**: 도구가 이미 존재 (Claude Code, git, npm 등). **프로세스 파이프라인** — "어떤 순서로 뭘 할지"만 안내하면 됨.
- **meta-pipe**: 도구 자체가 없을 수 있음. **프로세스 + 인프라 파이프라인** — 파이프라인 설계뿐 아니라 "이 단계를 실행할 도구"의 선택/세팅까지 다뤄야 함.

예시: 유튜브 섬네일 파이프라인을 만들어도, 이미지 AI를 MCP로 연결할지, API로 할지, 로컬 LLM을 쓸지 결정해야 하고, 결정한 뒤에도 세팅이 필요하다. 개발과 달리 "바로 쓸 수 있게" 되어 있지 않다. 게다가 API 사용 시 비용 문제도 있다.

### 핵심 인사이트

1. bkit의 개발 파이프라인도 "돌리면 뚝딱"이 아니라 PDCA 루프를 돌리며 개선한다. meta-pipe도 동일한 접근이 가능하다.
2. 다만 그 전에 **컨설팅(인터뷰) 단계**가 제대로 있어야 파이프라인 품질이 올라간다.
3. 이 컨설팅은 단순 질문-답변이 아니라, 사용자가 도메인 지식이 부족할 수 있으므로 AI가 조사해서 제안하고 조율하는 **컨설팅** 수준이어야 한다.

## 2. 확정된 방향

### 2.1 meta-pipe v2의 전체 흐름

```
Phase A: Consult (컨설팅)
  A1. 목표 인터뷰 — 뭘 만들고 싶은지, 어떤 결과를 원하는지
  A2. 환경/역량 확인 — 사용자 프로필 기반 (있으면 업데이트만)
  A3. 도구 조사 — AI가 자동으로 가능한 도구 리서치
  A4. 평가 기준 수립 — 사용자 취향 + 도메인 베스트 프랙티스 조율

Phase B: Discover (도메인 조사)
  — v1의 Phase A와 유사하나, 컨설팅 결과를 반영하여 조사 방향이 구체화됨

Phase C: Design Pipeline (파이프라인 설계)
  C1. Step 도출 — 목표 달성에 필요한 단계 설계
  C2. Step별 실행 모드 태깅 — auto / assist / manual
  C3. 도구 매핑 — 각 step에 어떤 도구를 쓸지
  C4. 세팅 필요 항목 정리

Phase D: Setup (도구 세팅) — 레벨에 따라 생략 가능
  D1. 필요한 도구 세팅 안내/실행

Phase E: Execute (실행)
  E1. 파이프라인 실행
  E2. step별 결과물 생성

Phase F: Evaluate & Improve (검수/개선)
  F1. 평가 기준 대비 결과물 검수
  F2. 기준 미달 시 자동/수동 개선 루프
```

### 2.2 Step별 실행 모드 태깅 (확정)

#### 논의 과정

파이프라인의 실행 주체를 누구로 할지 논의:

- 사용자의 최종 목표는 AI가 파이프라인을 자동 실행하는 것
- 하지만 현실적으로 기획/문서 작성은 AI가 가능하나, 이미지/영상 생성은 도구 세팅이 필요하고 비용 문제도 있음
- "사람용/AI용" 이분법 대신, **step마다 실행 주체가 다를 수 있다**는 인사이트 도출

예시 (유튜브 섬네일):

- 주제 분석, 레퍼런스 조사, 카피 작성 -&gt; AI 가능
- 이미지 생성 -&gt; 도구 세팅 여부에 따라 다름
- 최종 검수 -&gt; 아직은 사람 필요

#### 결정

파이프라인 전체를 "사람용/AI용"으로 나누는 게 아니라, 각 step에 실행 모드를 태깅:

```
execution_mode:
  - "auto"    → AI가 자동 실행 (API 호출 등)
  - "assist"  → AI가 초안 만들고 사람이 검수
  - "manual"  → 사람이 직접 실행, AI는 가이드만
```

실행 모드 결정 기준:

- 이 step은 자동화 가능한가?
- 필요한 도구가 세팅되어 있는가?
- 사용자가 직접 할 의향/역량이 있는가?

시간이 지나면서 manual -&gt; assist -&gt; auto로 점진적 업그레이드 가능.

### 2.3 복잡도별 레벨 분기 (확정)

bkit의 Starter/Dynamic/Enterprise 패턴 차용:

| 레벨 | 특징 | 예시 |
| --- | --- | --- |
| **Light** | 도구 세팅 불필요, 기존 도구로 충분 | 블로그 글쓰기 파이프라인 |
| **Standard** | 일부 도구 세팅 필요 | 유튜브 섬네일 (이미지 AI 세팅) |
| **Full** | 인프라 구축 수준의 세팅 필요 | 영상 자동 편집 파이프라인 |

레벨은 컨설팅 과정에서 자동 결정. 레벨에 따라 Phase D(Setup)의 범위가 달라짐.

### 2.4 평가 기준 (확정)

결과물 평가 기준은 두 소스의 조합:

- **사용자 기준**: 취향, 선호도, 주관적 품질 기준
- **도메인 기준**: AI가 도메인 조사에서 도출한 베스트 프랙티스

컨설팅 단계에서 이 둘을 조율. 단순 인터뷰가 아닌 **컨설팅**에 가까운 작업. AI의 자가 검수/개선 루프를 위해 가능한 한 정량적 기준으로 구체화.

**meta-pipe의 핵심 차별점**: "사용자가 원하는 결과를 AI가 스스로 검수할 수 있도록 기준을 구체화하는 과정"이 컨설팅에 포함된다. 정성적 기준("밝은 느낌")은 사람만 검수 가능하지만, 정량적 기준("텍스트 영역 30% 이상")은 AI도 검수 가능. 컨설팅에서 가능한 한 정량적 기준을 유도해내는 것이 자동화의 열쇠.

### 2.5 사용자 프로필 (확정)

환경/역량 인터뷰는 반복하지 않고, 프로필로 기록 -&gt; 업데이트 방식:

```
user-profile/
├── capabilities.md    # 사용자 역량 (API 세팅 가능, Docker 모름 등)
├── tools.md          # 현재 세팅된 도구들 (DALL-E API 키 있음 등)
└── preferences.md    # 선호도 (로컬 우선, 비용 최소화 등)
```

프로필이 쌓일수록 인터뷰가 짧아짐.

### 2.6 PDCA 적용 (확정)

PDCA 루프는 고정 단위가 아니라 유연하게 적용:

- 전체 파이프라인에 PDCA 사이클
- 필요 시 개별 step에도 별도 PDCA
- bkit의 PDCA 메커니즘 (gap-detector, pdca-iterator) 활용 가능

### 2.7 AI 자가 개선 (방향 확정)

최종 목표는 AI가 파이프라인을 실행하고 검수/개선까지 자동으로 수행하는 것. Harness 엔지니어링의 평가 기준(criteria) 기반 자가 생성/자가 개선 패턴 적용. 사용자 언급: "요즘 하네스 엔지니어링 자체도 AI의 자가 생성/자가 개선 쪽이니까, 검수와 개선까지도 나중엔 AI가 할 수 있다. 기준만 잘 세워진다면." ���, 현재는 도메인에 따라 사람 개입이 필요한 영역이 있음 -&gt; 실행 모드 태깅으로 현실과 타협하면서, 점진적으로 auto 비율을 높여가는 전략.

## 3. bkit에서 차용하는 패턴

| bkit 패턴 | meta-pipe 적용 |
| --- | --- |
| Starter/Dynamic/Enterprise 레벨 | Light/Standard/Full 레벨 |
| 각 Phase = 독립 스킬 | 각 Phase를 독립 스킬로 분리 |
| 필요 시점에 인프라 세팅 (Phase 4 진입 시 백엔드 등) | step 실행 시 도구 없으면 그때 세팅 안내 |
| PDCA 사이클 | 동일 적용 (전체/step 단위 모두 가능) |
| gap-detector + pdca-iterator | 결과물 평가/개선 루프에 활용 |
| 레벨 자동 감지 (프로젝트 구조 분석) | 컨설팅에서 레벨 자동 결정 |
| 9단계이나 레벨에 따라 필요한 단계만 사용 | 풀 파이프라인(A\~F) 있되 레벨에 따라 간소화 |
| PreToolUse/PostToolUse 훅으로 자동 PDCA | 향후 자동화 확장 시 참고 |
