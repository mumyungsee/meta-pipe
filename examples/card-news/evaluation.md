## 3. 파이프라인 도메인 적합성

### 카드뉴스 도메인에 특화된 단계인가?

| 검증 항목 | 결과 | 설명 |
| --- | --- | --- |
| 범용 소프트웨어 개발과 차별화 | **예** | Step 3(템플릿 엔진), Step 4(콘텐츠 파이프라인), Step 7(접근성/법적 준수)은 카드뉴스 고유 |
| 도메인 용어가 단계에 반영 | **예** | 슬라이드, 커버, 톤앤매너, 판형 등 도메인 용어가 각 단계에 녹아 있음 |
| 도메인 리스크가 반영 | **예** | 저작권(Step 7), 접근성(Step 7), 한글 렌더링(Step 5)이 명시적 |
| 불필요한 단계 존재 | **아니오** | 8개 단계 모두 카드뉴스 자동화에 필수적 |
| 누락된 단계 존재 | **일부** | "배포 자동화" 단계가 독립 step으로 분리될 수 있으나, Step 6에 포함 처리 |

### bkit 9단계와 비교

```
bkit (웹앱 고정)              meta-pipe (카드뉴스 맞춤)
───────────────────────────   ─────────────────────────────
1. 프로젝트 정의               1. 도메인 용어 & 요구사항 정의
2. 기능 분석                   2. 데이터 모델 설계 (카드뉴스 엔티티)
3. 목업/와이어프레임 ← 불필요   3. 템플릿 엔진 설계 ← 카드뉴스 고유
4. API 설계 ← 불필요           4. 콘텐츠 파이프라인 설계 ← 카드뉴스 고유
5. 디자인 시스템                5. 렌더링 엔진 구현
6. UI 구현 ← 불필요            6. 플랫폼 어댑터 & 내보내기
7. 백엔드 구현                  7. 접근성 & 법적 준수 ← 카드뉴스 고유
8. 테스트                      8. 통합 테스트 & 품질 검증
9. 배포                        (Step 6에 포함)
```

bkit의 9단계 중 3개(목업, API 설계, UI 구현)가 카드뉴스에 불필요하고, 대신 3개의 도메인 특화 단계(템플릿 엔진, 콘텐츠 파이프라인, 접근성/법적 준수)가 추가되었다. **이것이 meta-pipe의 핵심 가치를 증명한다.**

## 4. 전문가 수준 여부

| 평가 항목 | 점수 (1-5) | 근거 |
| --- | --- | --- |
| 도메인 이해도 | **4/5** | 용어, 판형, 워크플로우, 법적 리스크까지 파악. 다만 실무 제작 경험에서 오는 암묵지는 부족 |
| 기술 스택 적절성 | **5/5** | HTML/CSS + Puppeteer 방식은 업계에서 검증된 접근법. node-html-to-image, Sharp 등 실무 도구 선정 |
| 데이터 모델 완성도 | **4/5** | 핵심 엔티티와 관계 도출 완료. 실제 구현 시 필드 추가/수정 필요할 수 있음 |
| 리스크 식별 | **5/5** | 저작권, 접근성, 한글 렌더링 등 카드뉴스 도메인 핵심 리스크 모두 식별 |
| 파이프라인 실현 가능성 | **4/5** | 각 단계의 산출물과 완료 기준이 구체적. 콘텐츠 파이프라인(Step 4)의 LLM 프롬프트 품질은 실험 필요 |

**종합 점수: 4.4/5** - 카드뉴스 전문가가 보기에 납득 가능한 수준이며, 실제 구현에 바로 착수할 수 있는 수준의 파이프라인.

### 개선 제안

1. **WebFetch로 KISS 논문** 심층 분석 -&gt; 데이터 모델/아키텍처 보완
2. **콘텐츠 파이프라인 프롬프트** 프로토타이핑 -&gt; 실제 LLM 출력 품질 검증
3. **기존 SaaS(Tyle.io, Canva) 실사용** -&gt; 템플릿 패턴 벤치마크
4. **한글 폰트 렌더링 PoC** -&gt; Puppeteer + 한글 폰트 조합 검증
